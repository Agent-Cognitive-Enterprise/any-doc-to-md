# 2026-05-27 — Paragraph Continuity Repair Plan

**Branch:** `paragraph-continuity-repair`  
**Repository:** `Agent-Cognitive-Enterprise/any-doc-to-md`  
**Status:** Planning document for SABRE implementation  
**Primary constraint:** ADTM is live and open source. Backward compatibility is mandatory.

---

## 1. Problem statement

Some converted documents arrive in Markdown with each visual row or extracted text line represented as a separate paragraph. Downstream ADTM consumers may then chunk by paragraph and extract claims from paragraph-sized units. When prose has been shredded into row-sized fragments, chunking and claim extraction become noisy, incomplete, and misleading.

This issue should be handled before chunking and before claim extraction. In ADTM terms, it belongs in the conversion-normalisation/repair layer, not in downstream retrieval or claim logic.

Goal: add a deterministic, fast, non-LLM paragraph continuity repair path that can:

1. detect likely row-sliced prose,
2. merge only high-confidence continuation runs,
3. avoid headings, tables, lists, code blocks, images, captions, and other structural Markdown,
4. accept the repaired output only when a quality gate shows clear improvement,
5. preserve original adapter output for inspection,
6. preserve existing CLI/API/default behavior compatibility.

## 3. Compatibility guardrails

These are hard constraints for every slice.

### 3.1 No breaking defaults

Do not change default CLI behavior in a way that surprises current users.

Allowed:

- a built-in repair that only changes final `index.md` when a strict quality gate accepts the repair,
- additive metadata in `anydoc2md-result.json` if carefully documented,
- optional CLI/config controls with default-compatible behavior.

Not allowed without explicit approval:

- removing or overwriting raw adapter `index.md`,
- changing required CLI arguments,
- requiring LLM, network, optional converter binaries, or new heavyweight dependencies,
- changing default adapter list,
- changing existing public dataclass constructor arguments in a breaking way,
- changing `run_full_tournament(...)` positional call compatibility,
- replacing current fix-extension behavior.

### 3.2 No dependency bloat

Prefer Python standard library only. Do not add NLP packages, sentence tokenizers, OCR tools, or model-backed detectors for this feature.

### 3.3 Deterministic and fast

Repair must be cheap enough for normal conversion. It should operate over Markdown text with linear or near-linear complexity.

### 3.4 Conservative Markdown handling

Never merge across:

- headings,
- list items,
- table-looking lines,
- fenced code blocks,
- indented code blocks,
- blockquotes unless explicitly supported and tested,
- image lines,
- figure/table captions,
- horizontal rules,
- raw HTML blocks,
- front matter boundaries,
- blank-line-separated sections where the detector is uncertain.

### 3.5 Evidence and traceability

Repair should return structured evidence:

- whether repair was attempted,
- whether it was accepted,
- adapter name when repair is applied to a staging directory,
- source path or document key when repair is applied to a staging directory,
- adapter staging path when repair is applied to a staging directory,
- original paragraph count,
- repaired paragraph count,
- number of merge groups,
- before/after quality summary,
- reasons/signals that triggered or rejected repair,
- bounded examples only, not full-document dumps.

Examples must be bounded by both count and size. Prefer line or paragraph
references plus short snippets over copied prose. Sidecars must not contain a
full source document, full candidate Markdown, large paragraphs, private source
excerpts, or unbounded evidence dumps.

### 3.6 Staging and fixed-output ownership

`index_fixed.md` is a generated improved-output candidate, not raw adapter
output. Current tournament code creates adapter directories but does not
guarantee that every old generated file is removed before a new conversion.
Because selector and publisher prefer `index_fixed.md` when present, stale
fixed output is a direct compatibility risk.

Before any orchestration slice trusts or preserves `index_fixed.md`, it must
establish current-run ownership.

Required rules:

- raw adapter `index.md` remains the source of truth after each adapter run,
- stale `index_fixed.md` from a previous run must not be selected, scored, or
  published for a new run,
- a pre-existing `index_fixed.md` may be used as a fix-extension base only when
  it is known to belong to the current run, for example because the built-in
  repair just wrote it and a sidecar/manifest matches the current raw
  `index.md`,
- dirty staging directories must be tested explicitly, including reruns to the
  same CLI output path,
- cleanup must never delete raw adapter `index.md`, images, adapter result JSON,
  project-local scaffold files, or unrelated user files outside the adapter
  staging directory.

### 4.3 Detection strategy

Compute document-level and run-level signals over prose-like Markdown paragraphs.

Signals:

- median paragraph length,
- short paragraph ratio,
- no-terminal-punctuation ratio,
- lowercase-start ratio,
- continuation-pair ratio,
- runs of 4+ short continuation-like paragraphs,
- suspiciously uniform paragraph lengths,
- proportion of paragraphs excluded as structure.

A document/run should be repairable only when enough signals agree. Short paragraphs alone are not enough.

### 4.4 Merge strategy

Merge adjacent prose blocks only when `should_merge(a, b)` is true and no hard boundary sits between them.

Positive merge signals:

- previous paragraph lacks terminal punctuation,
- previous paragraph ends with comma, semicolon, colon, hyphen, dash, or opening bracket,
- previous paragraph ends with continuation words such as `and`, `or`, `of`, `to`, `in`, `with`, `for`, `by`, `from`, `that`, `which`,
- next paragraph starts lowercase,
- both paragraphs are short,
- joined text remains under `max_merged_paragraph_chars`.

Negative/hard-break signals:

- either side is structural Markdown,
- previous paragraph ends terminal and next starts uppercase,
- next paragraph is heading/list/table/caption/code/image/raw HTML,
- merged paragraph would be too long,
- blank boundary is semantically meaningful and confidence is low.

Hyphenated line endings should be joined without an extra space only for safe word continuations. Be conservative.

### 4.5 Quality gate

Repair should be accepted only when quality improves meaningfully.

Before/after quality should consider:

- fewer incomplete prose paragraphs,
- fewer very short prose paragraphs,
- more complete sentence-like paragraphs,
- paragraph count reduction without collapsing the document into giant blobs,
- no increase in structural Markdown damage,
- no paragraph over configured max size,
- text content preserved after whitespace normalization.

Quality gate must verify normalized textual content is preserved. It may alter whitespace and hyphenation, but must not drop words.

### 4.6 Interaction with existing fix extensions

Important implementation decision for builder/reviewer:

Current `apply_fix_extensions(...)` applies project-local fix extensions and writes `index_fixed.md` if the extension improves QA score.

The built-in paragraph repair must not trample project-local fixes. Two acceptable designs:

#### Design A — integrated built-in repair before project-local fixes

1. start from `index.md`,
2. run paragraph repair and keep candidate in memory if accepted,
3. then run project-local fix extensions against the current best text,
4. write final `index_fixed.md` if any built-in or project-local fix improved/was accepted.

Pros: one final fixed output.  
Cons: requires careful modification of `fix_application.py`.

#### Design B — separate pre-fix built-in repair step

1. orchestrator calls built-in paragraph repair before `apply_fix_extensions(...)`,
2. repair writes `index_fixed.md` if accepted,
3. `apply_fix_extensions(...)` must be updated to start from `index_fixed.md` when it exists instead of always `index.md`, while still restoring `index.md` unchanged.

Pros: clear separation.  
Cons: requires changing fix-extension base-text behavior.

Preferred: **Design B**, because it keeps repair as a focused built-in normalization stage and lets existing fix extensions continue improving the best current candidate.

Backward compatibility requirement for Design B:

- when no `index_fixed.md` exists, `apply_fix_extensions(...)` behavior must be identical to current behavior,
- when a current-run `index_fixed.md` exists, project-local fix extensions should improve from that repaired candidate, not discard it,
- stale `index_fixed.md` handling must not delete an accepted current-run built-in repair just because no project-local fix improves it,
- arbitrary old `index_fixed.md` files in dirty staging dirs must be ignored or removed before selection.

This is subtle. Reviewer should red-team this hard.

### 4.8 Suggested detection approach

Detect broken paragraphs with multiple cheap deterministic signals. Short
paragraphs alone must not trigger repair.

Compute signals over prose-only Markdown blocks:

- `short_ratio`: share of prose blocks below a conservative length threshold,
- `no_terminal_ratio`: share of prose blocks without sentence-ending
  punctuation,
- `continuation_pair_ratio`: share of adjacent prose pairs that look joinable,
- `lowercase_start_ratio`: share of next prose blocks starting lowercase after a
  non-terminal previous block,
- `qualifying_continuation_run_count`: count of runs meeting
  `settings.min_continuation_run_blocks`,
- `longest_continuation_run`: longest run of continuation-like prose blocks,
- `structure_ratio`: share of non-blank content blocks excluded as headings,
  lists, tables, code, captions, images, blockquotes, HTML, or front matter.

Suggested pair-level heuristic:

```python
def looks_like_continuation(
    left: str,
    right: str,
    settings: ParagraphRepairSettings,
) -> bool:
    if not left.strip() or not right.strip():
        return False
    if ends_terminal(left):
        return False
    if starts_structural_markdown(right):
        return False
    if max(len(left), len(right)) < settings.min_continuation_chars:
        return False

    length_signals = 0
    length_signals += len(left) <= settings.short_prose_chars
    length_signals += len(right) <= settings.short_prose_chars

    continuation_signals = 0
    continuation_signals += starts_lowercase(right)
    continuation_signals += ends_with_continuation_word(left)
    continuation_signals += left.rstrip().endswith((",", ";", ":", "-", "(", "["))
    return length_signals >= 1 and continuation_signals >= 1
```

Suggested initial document-level threshold shape, in reason-code order:

```text
prose_blocks >= settings.min_paragraphs
short_ratio >= settings.min_short_ratio
no_terminal_ratio >= settings.min_no_terminal_ratio
continuation_pair_ratio >= settings.min_continuation_ratio
lowercase_start_ratio >= settings.min_lowercase_start_ratio
structure_ratio <= settings.max_structure_ratio
longest_continuation_run >= settings.min_continuation_run_blocks
```

Run-level evidence should also be reported so a mixed document with one
row-sliced section can be repaired later without requiring the whole document
to look fragmented.

Initial detector scope is Markdown-paragraph fragmentation: visual rows that
became separate Markdown paragraph blocks. Soft line breaks inside one Markdown
paragraph should be documented and tested as a non-trigger unless a later slice
explicitly expands scope, because Markdown semantics already treat those lines
as one paragraph.

## 8. Known risks

| Risk | Mitigation |
|---|---|
| False positive repair on intentional short prose, poetry, lyrics, legal clauses, or checklists | Require multiple signals, exclude lists/tables/headings, add opt-out if needed |
| Table rows merged as prose | Conservative table-like classifier and structural-count quality gate |
| Code fences altered | Dedicated block parser and byte-preservation tests |
| Current-run `index_fixed.md` deleted by fix application | Slice 9 explicitly preserves and tests trusted current-run fixed output |
| Stale `index_fixed.md` from dirty staging dirs selected for a new run | Slice 8A clears or ignores unowned fixed-output artifacts before repair, fix extension, selection, and promotion |
| Score changes alter adapter winner | Keep fragmentation QA warning modest and document intended quality effect |
| Public API break from dataclass fields | Avoid changing public constructors or add defaulted keyword-only fields only |
| Dependency creep | Standard library only |
| Sidecar report leaks too much document text | Bound example count and size, prefer references over snippets, and test sidecar size/content |
| Sidecar report becomes accidental contract | Keep sidecar additive, documented as diagnostic, and not required by selector |
