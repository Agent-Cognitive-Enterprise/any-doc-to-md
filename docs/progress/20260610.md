## Slice 10 Review Fix: Make Default Paragraph Repair Observable

**Status:** complete  
**Date:** 2026-06-10  
**Role:** build after red-team review

**Review finding addressed:**

The first Slice 10 wiring was mechanically correct, but the default
`ParagraphRepairSettings.min_paragraphs=20` meant realistic short row-sliced
documents were rejected before the evidence gates ran. That created a
claim/reality gap: default repair was documented and wired but hard to observe
through the real CLI or public benchmark workflow.

**Implemented changes:**

- Lowered the default paragraph-repair document-size floor from 20 prose blocks
  to 8.
- Kept the semantic gates unchanged: structure ratio, short-fragment ratio,
  no-terminal ratio, lowercase-start ratio, continuation-pair ratio,
  continuation-run length, content preservation, structural-count preservation,
  and quality delta still all gate acceptance.
- Added `examples/benchmark-corpus/row-sliced-note.txt`, a small package-owned
  public fixture that demonstrates the default repair path through the real
  in-house adapter and tournament pipeline.
- Added a narrow `.gitignore` unignore for `examples/benchmark-corpus/` because
  the existing `benchmark-*/` ignore pattern hid new package-owned public
  fixture files.
- Updated the public benchmark reproduction guide for the four-fixture current
  smoke and documented that the older dated public-fixture snapshot remains a
  historical three-document measurement.
- Changed README wording from "current" to "dated" for the 2026-04-27 public
  fixture snapshot.

**Tests added/updated:**

- Detector tests now prove a 13-fragment row-sliced document is detected by
  default and a 7-fragment version remains below the floor.
- Application tests now prove the same realistic 13-fragment document is
  accepted and repaired in memory.
- CLI tests now run a real conversion twice, default `auto` and explicit `off`,
  and assert only the default path publishes joined Markdown plus winner
  paragraph-repair artifacts.
- Public benchmark tests now include `row-sliced-note.txt` and assert the real
  tournament writes `index_fixed.md`, `index_paragraph_repaired.md`, and
  `paragraph_repair_report.json` for that fixture.

**Implemented changes:**

  check.
- Reused the built-in paragraph repair detector instead of duplicating
  fragmentation heuristics.
- The check returns `warn` only when the detector reports row-sliced prose; it

  eligible row-sliced adapter by score.
- Updated README, spec, and troubleshooting docs for the new warning and the
  actual repair/fix/QA ordering.

**Behavior change (intended):**

- Unrepaired row-sliced Markdown is now visible in QA reports and less likely to
  win selection.
- Accepted paragraph repair normally removes the warning because selection and
  publishing score `index_fixed.md` when present.
- The warning is not a hard gate and does not disqualify a candidate.

**Tests added/updated:**

- Selector integration test covers the scoring effect against real staging dirs.

## Slice 12 Review Fix: Explicit Document-Level Fragmentation Score

**Status:** complete
**Date:** 2026-06-10
**Role:** build after red-team review

**Review findings addressed:**

- The first Slice 12 implementation accidentally tied the
  `paragraph_not_row_sliced` score to the number of diagnostic detail strings.
- Tests asserted fixture-specific signal values and hardcoded the detail-count
  multiplier.
- The QA check called the detector with hardwired default settings.
- Docs did not acknowledge that the detector's lowercase/continuation-word
  heuristics are Latin-script biased.

**Implemented changes:**

- Added `DOCUMENT_LEVEL_CHECK_MULTIPLIERS` in `output_qa/scoring.py` and routed
  `paragraph_not_row_sliced` through an explicit fixed multiplier of `2`.
  Diagnostic detail formatting can no longer change the score.
- Kept the current 6-point warning penalty intentionally:
  `1.5 check weight * 2.0 warn severity * 2 document-level multiplier`.
- Added `_violation_multiplier(...)` to keep ordinary per-detail checks using
  the existing capped detail-count behavior.
- Added optional `settings: ParagraphRepairSettings | None` to
  `check_paragraph_not_row_sliced(...)` so future callers can align QA and
  repair thresholds without changing the function shape.
- Loosened paragraph-fragmentation tests away from exact fixture signal values;
  tests now assert signal keys, bounded/no-excerpt details, explicit
  document-level score behavior, scorecard integration, and custom settings.
- Updated spec and troubleshooting docs to state that the warning is
  language-uneven because the detector uses Latin-script lowercase and
  continuation-word heuristics.

## Slice 12 Review Fix 2: Repair-Mode-Independent Fragmentation Warning

**Status:** complete  
**Date:** 2026-06-10  
**Role:** build after red-team review

**Review finding addressed:**

- The `settings` parameter added to `check_paragraph_not_row_sliced(...)` was
  untyped and inert in the real pipeline: `run_all(...)` calls the check with no
  settings and has no settings parameter, and the selector passes none. The
  parameter was reachable only from unit tests, which implied an end-to-end
  configurability that did not exist.
- The interaction with `--paragraph-repair off` was unspecified and untested. In
  off mode the check still warns (it uses default `enabled=True`), so disabling
  repair does not silence the penalty — defensible, but neither documented nor
  pinned by a test.

**Resolution (design intent):**

The QA warning is a quality-visibility signal and is intentionally independent
of the repair mode. `off` disables the auto-fix, not the signal; repair clears
the warning only by producing a clean `index_fixed.md` that selection scores.
Threading `enabled=False` into QA was rejected because it would let a fragmented
adapter win silently in off mode.

**Implemented changes:**

- Typed the override as `settings: ParagraphRepairSettings | None = None` and
  documented that the tournament always calls with defaults, that the warning is
  orthogonal to `--paragraph-repair`, and that `settings` must not be used to
  gate the warning on whether repair ran.
- Added `test_warning_is_independent_of_repair_mode` pinning that the default
  (pipeline) call warns on row-sliced prose while only an explicit
  `enabled=False` override suppresses it.
- Stated the repair-mode independence in the tournament spec and troubleshooting
  docs.
- Reverted the collateral trailing-whitespace edit that had stripped the
  Markdown hard breaks from the two pre-existing Slice 10 progress entries.
