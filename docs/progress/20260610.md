## Slice 12 Review Fix 2: Repair-Mode-Independent Fragmentation Warning

**Status:** complete  
**Date:** 2026-06-10  
**Role:** build after red-team review

**Review finding addressed:**

- The `settings` parameter added to `check_paragraph_not_row_sliced(...)` was
  untyped and inert in the real pipeline: `run_all(...)` calls the check with no
  settings and has no settings parameter, and the selector passes none. The
  parameter was reachable only from unit tests, which implied an end-to-end
  configurability that did not exist.
- The interaction with `--paragraph-repair off` was unspecified and untested. In
  off mode the check still warns (it uses default `enabled=True`), so disabling
  repair does not silence the penalty — defensible, but neither documented nor
  pinned by a test.

**Resolution (design intent):**

The QA warning is a quality-visibility signal and is intentionally independent
of the repair mode. `off` disables the auto-fix, not the signal; repair clears
the warning only by producing a clean `index_fixed.md` that selection scores.
Threading `enabled=False` into QA was rejected because it would let a fragmented
adapter win silently in off mode.

**Implemented changes:**

- Typed the override as `settings: ParagraphRepairSettings | None = None` and
  documented that the tournament always calls with defaults, that the warning is
  orthogonal to `--paragraph-repair`, and that `settings` must not be used to
  gate the warning on whether repair ran.
- Added `test_warning_is_independent_of_repair_mode` pinning that the default
  (pipeline) call warns on row-sliced prose while only an explicit
  `enabled=False` override suppresses it.
- Stated the repair-mode independence in the tournament spec and troubleshooting
  docs.
- Reverted the collateral trailing-whitespace edit that had stripped the
  Markdown hard breaks from the two pre-existing Slice 10 progress entries.

## Slice 16: Branch Closure Hardening

**Status:** complete
**Date:** 2026-06-10
**Role:** build after branch-level red-team review

**Objective:**

Close residual robustness gaps found during branch-level review before treating
the paragraph-continuity-repair branch as goal-complete.

**Review findings addressed:**

- `apply_paragraph_continuity_repair(...)` promised to clear helper-owned stale
  artifacts, but the direct helper cleanup used `Path.unlink()` and would raise
  if a stale `index_paragraph_repaired.md` or `paragraph_repair_report.json`
  path was a directory. The orchestrator hygiene guard masked common paths, but
  the helper itself was not robust.
- `run_tournament(...)` used `ThreadPoolExecutor` as a context manager around
  the wall-clock timeout guard. On `TimeoutError`, the context manager still
  waited for worker shutdown, so a truly hung adapter thread could block the
  function instead of returning a timeout result.

**Implemented changes:**

- Made paragraph-repair owned-artifact cleanup type-robust for files, symlinks,
  and directories.
- Added a regression proving stale owned artifact directories are cleared on a
  missing-`index.md` direct helper call.
- Reworked tournament runner executor shutdown so timeout paths call
  `shutdown(wait=False, cancel_futures=True)`.
- Added a regression proving `run_tournament(...)` returns a timeout result
  before a blocked adapter releases.

**Residual limitation:**

- Python threads cannot be killed safely, so a timed-out adapter thread keeps
  running until its own code returns. Returning without joining it has two
  residual consequences, not zero:
  1. The post-timeout `clear_failed_adapter_output(...)` now runs concurrently
     with the still-live worker, so a late write can leave artifacts in the
     staging slot. Those artifacts cannot win the current run — the name's
     result is `status="timeout"` — and `prepare_adapter_run_output_slot(...)`
     re-clears the slot before the next reuse, so the exposure is bounded.
  2. The worker is a non-daemon `concurrent.futures` thread, so a never-returning
     adapter is still joined by the interpreter's atexit handler and can block
     process shutdown even though `run_tournament(...)` itself returned promptly.
  The function no longer blocks on a hung worker, but "does not block forever"
  applies to the call, not the process. Full late-write isolation would require a
  process-level adapter execution boundary.
